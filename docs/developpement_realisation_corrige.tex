\chapter{Développement et Réalisation}

\section{Introduction}

La phase de développement et de réalisation constitue l'aboutissement technique de notre projet de détection du churn pour Tunisie Telecom. Dans ce chapitre, nous détaillons les choix d'implémentation et les mécanismes logiciels que nous avons mis en oeuvre pour transformer nos modèles théoriques en une plateforme opérationnelle.

Conformément à la méthodologie Scrum que nous avons adoptée, nous présentons notre travail de réalisation sous la forme d'incréments successifs correspondant aux trois derniers sprints du projet. Cette structure nous permet de démontrer la traçabilité entre les objectifs fixés dans le backlog, les choix techniques de réalisation, les livrables produits sous forme de releases et leur validation par les tests.

\section{Environnement de développement et outils}

Avant de détailler les sprints, nous présentons le socle technologique commun à l'ensemble du projet. Nous utilisons Python 3.12 comme langage de base. Le backend repose sur Django 4.2.30, tandis que le service d'intelligence artificielle est porté par FastAPI 0.115.0. La persistance des données est assurée par PostgreSQL 15, et l'explicabilité du modèle s'appuie sur la bibliothèque SHAP 0.51.0. Nous avons organisé notre code source sous Git en suivant une structure modulaire composée de plusieurs applications Django (accounts, learning, dashboard et core).

\subsection{Réalisation technique}

Nous avons implémenté l'API d'inférence en utilisant FastAPI pour ses capacités asynchrones. Nous avons adopté le pattern Lifespan pour charger les artefacts ML (modèle Random Forest et explainer SHAP) une seule fois au démarrage. Pour la sécurité, nous avons développé un moteur OTP intégré. Lorsqu'un utilisateur saisit ses identifiants, nous générons un code unique envoyé par email via SendGrid, avec une validité de cinq minutes. Nous avons également intégré un mécanisme de verrouillage qui bloque le compte après trois tentatives infructueuses.

\subsection{Livrable et Release du Sprint 4}

La release de ce sprint fournit un système d'accès sécurisé et un service de prédiction fonctionnel. L'interface de validation OTP que nous avons conçue constitue le point d'entrée principal pour les utilisateurs.

\section{Sprint 5 : Intégration des données et Pipeline ML}

\subsection{Objectif du sprint}

\subsection{Réalisation technique}

Nous avons conçu le modèle ClientChurn pour regrouper plus de quarante variables comportementales. Pour garantir l'isolation, nous avons mis en place un middleware nommé AgenceMiddleware qui filtre systématiquement les requêtes ORM. Nous avons également développé un service de communication asynchrone entre Django et FastAPI. Pour assurer la continuité de service, nous avons intégré un pipeline de failover : si FastAPI est injoignable, Django bascule automatiquement sur un modèle ML local pour fournir les prédictions de base.

\subsection{Réalisation technique}

Nous avons personnalisé l'expérience utilisateur selon les profils : le Chef d'Agence dispose d'une vue stratégique tandis que les agents accèdent à une liste opérationnelle. Nous avons intégré les graphiques SHAP Waterfall directement dans la fiche client pour expliquer les raisons du risque. Nous avons également implémenté un workflow de recommandation en six étapes avec un centre de notifications internes. Enfin, nous avons utilisé pdfkit pour générer des rapports professionnels incluant les graphiques d'explicabilité.

Un effort particulier a été porté sur le \textbf{Centre de Pilotage des Recommandations}, qui matérialise la stratégie commerciale :
\begin{itemize}
    \item \textbf{Identification continue :} Nous avons implémenté une logique d'\texttt{update\_or\_create} lors des imports de datasets. Cela garantit que le système reconnaît le client à chaque analyse et conserve l'historique complet de ses actions passées, assurant une mémoire commerciale sans faille.
    \item \textbf{Pilotage par la Valeur :} L'interface affiche désormais l'impact financier estimé (Odt/TND) pour chaque action, permettant une exécution basée sur le ROI.
    \item \textbf{Rôles Métiers :} L'accès au centre de pilotage est strictement restreint aux profils opérationnels (Chef, Marketing, Commercial), renforçant la confidentialité et la spécialisation des tâches.
\end{itemize}

\section{Bilan de la réalisation et Perspectives}

\subsection{Orchestration et déploiement}

Nous avons achevé ce cycle de développement par l'orchestration de nos services via Docker Compose. Cette approche nous permet de garantir un environnement identique entre le développement et la production chez Tunisie Telecom, en pilotant simultanément Django, FastAPI et PostgreSQL.

\subsection{Limites identifiées}

Malgré la complétude de notre solution, nous notons que le calcul des valeurs SHAP peut devenir gourmand en mémoire lors d'analyses massives. De plus, notre système ne dispose pas encore d'une boucle de rétroaction automatique permettant de mesurer le succès réel des recommandations sur le terrain.

\section{Conclusion}

Ce chapitre a permis de présenter la réalisation concrète de notre plateforme à travers ses incréments successifs. Nous avons démontré comment chaque sprint a contribué à construire une solution robuste, sécurisée et parfaitement adaptée aux besoins opérationnels de Tunisie Telecom. L'association de Django et FastAPI nous a permis de livrer un outil d'aide à la décision complet, prêt pour une exploitation en agence. Il est important de renseigner que ces tests utilisent un dataset synthétique comportant les mêmes features que celui qui nous a permis de mener nos travaux de Machine Learning.
